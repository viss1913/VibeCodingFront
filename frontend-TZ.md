## ТЗ для фронтенд‑разработчика проекта «Вайб Кодинг для Руководителей»

### 1. Цель проекта

Сделать современный, быстрый и понятный веб‑интерфейс для:
- продажи и презентации курса «Вайб Кодинг для Руководителей»;
- личного кабинета ученика с доступом к урокам, ДЗ и прогрессу;
- (опция) панели администратора для управления студентами и контентом.

Фронт должен работать как отдельное SPA/SSR‑приложение и общаться с бекендом по REST/HTTP (JSON).

---

### 2. Технологический стек (желаемый)

- **Фреймворк:** React + Next.js (SSR/SSG для SEO и скорости).
- **Язык:** TypeScript.
- **Стили:** Tailwind CSS или CSS‑модули (по согласованию).
- **Маршрутизация:** Next.js `app` или `pages` router.
- **Сборка/деплой:** Vercel.
- **Формы:** React Hook Form (или аналог).
- **Состояние:** минимально через React Context/Zustand (без тяжелого Redux, если не нужен).

Если предлагаешь другой стек — прописать, чем лучше и что меняется.

---

### 3. Страницы и маршруты

#### 3.1. Публичные страницы

1. **Главная / лендинг курса** — `/`
   - Hero‑блок: название курса, оффер, CTA‑кнопки «Записаться», «Программа».
   - Кратко: для кого курс, какие боли решает.
   - Блок «Что вы получите» (результаты после курса).
   - Блок «Программа курса» (свёрнутый список уроков с возможностью раскрыть).
   - Блок «Формат и стоимость» (цены, тарифы, рассрочка).
   - Блок «Отзывы» (подготовить структуру под реальные отзывы).
   - FAQ (взять из `FAQ.md` из бека как контент‑ориентир).
   - Футер: контакты, политика конфиденциальности, оферта.

2. **Страница программы курса** — `/program`
   - Детальный список уроков с описаниями (ориентироваться на `course-structure.md` из бека).
   - Для каждого урока: номер, название, цель, краткое содержание, формат, предполагаемое ДЗ.
   - Якорные ссылки на каждый урок (`#lesson-1`, `#lesson-2`, ...).

3. **Страница тарифов/оплаты** — `/pricing`
   - Список тарифов (Базовый / Расширенный / Премиум — ориентир из `pricing-analysis.md`).
   - Для каждого тарифа: цена, что входит, длительность, CTA‑кнопка «Выбрать».
   - После выбора тарифа фронт должен вызвать бекенд‑endpoint для создания сессии оплаты (Stripe/ЮKassa) и сделать редирект.
   - Страницы состояний:
     - `/payment/success` — успешная оплата;
     - `/payment/cancel` — отмена/ошибка.

4. **Статические инфо‑страницы**
   - `/privacy` — Политика конфиденциальности.
   - `/offer` — Публичная оферта / условия.

#### 3.2. Аутентификация

Маршруты:
- `/auth/login` — вход;
- `/auth/register` — регистрация;
- `/auth/forgot` — восстановление пароля;
- `/auth/reset?token=...` — установка нового пароля.

Требования:
- Валидация форм на фронте + отображение ошибок от бека.
- Сохранение access‑токена (JWT) **только** в `httpOnly` cookie (токен отдает бекенд; фронт не хранит его в LocalStorage).
- Хранить в клиентском стейте только минимальную информацию о пользователе (id, имя, роль).

#### 3.3. Личный кабинет ученика

Базовый путь: `/app/*` (закрытые роуты, доступны только авторизованным).

1. **Дашборд** — `/app`
   - Приветствие по имени.
   - Прогресс по курсу: завершено уроков / всего, % прогресса.
   - Ближайшие/рекомендуемые шаги (следующий урок, невыполненные ДЗ).

2. **Список уроков** — `/app/lessons`
   - Табличка/карточки по урокам:
     - номер, название, статус (не начат / в процессе / завершен);
     - отметка о выполнении ДЗ (если бекенд отдает).
   - Клик по уроку → переход на страницу урока.

3. **Страница урока** — `/app/lessons/[id]`
   - Заголовок, цель урока, краткое содержание.
   - Блок «Домашнее задание» (контент по аналогии с `homework-assignments.md`).
   - Блок для ссылки на видео/материал (URL придет с бека).
   - Кнопка «Отметить как выполнено» / «Отправить ДЗ» (вызов бекенд‑API).

4. **Профиль пользователя** — `/app/profile`
   - Имя, email, аватар (опционально).
   - Настройки уведомлений (email/Telegram — пока только UI, бекенд‑интеграция по готовности).
   - Кнопка «Выйти».

5. **(Опция) Админ‑панель** — `/app/admin/*`
   - Доступ только для роли `admin`.
   - CRUD по пользователям, урокам, ДЗ — **пока можно ограничиться заготовками интерфейса** и мок‑данными.

### 4. Взаимодействие с бекендом

Бекенд находится в отдельном репозитории (`back vibe coding`). Конкретные URL и схемы ответов будут предоставлены позже. Пока требуется:

1. **Сервис API‑клиента**
   - Общий модуль для запросов (axios/fetch wrapper).
   - Базовый URL брать из `process.env.NEXT_PUBLIC_API_URL`.
   - Обработка ошибок (toast/уведомления).

2. **Примерные сущности (ожидание по данным от бека)**
   - `User`: `{ id, name, email, role }`
   - `Lesson`: `{ id, number, title, goal, shortDescription, status, hasHomework }`
   - `HomeworkStatus`: `{ lessonId, status, submittedAt, grade? }`
   - `Tariff`: `{ id, name, price, features[] }`

3. **Авторизация**
   - Фронт вызывает `/auth/login` → бекенд ставит `httpOnly` cookie.
   - Для проверки сессии фронт дергает `/auth/me` и получает профиль пользователя.
   - Закрытые страницы делают редирект на `/auth/login`, если пользователь не авторизован.

4. **Оплата**
   - Фронт отправляет на бекенд данные выбранного тарифа.
   - Бекенд возвращает ссылку/ID на платежную сессию → фронт редиректит пользователя.
   - После возврата с платежной системы фронт читает query‑параметры и показывает успешный/ошибочный экран.

### 5. Дизайн и UX

#### 5.1. Общие требования
- Современный минималистичный дизайн (ориентир: сайты Notion, Linear, Vercel).
- Цветовая палитра: светлая тема + (опция) темная.
- Адаптивность: мобильные, планшеты, десктопы.
- Четкая иерархия заголовков, много воздуха, крупные CTA‑кнопки.

#### 5.2. Компоненты
Реализовать набор базовых UI‑компонентов:
- Кнопки (`Button`) — primary, secondary, outline, disabled.
- Инпуты/формы (`Input`, `Textarea`, `Select`, `Checkbox`, `Radio`).
- Навигация (`Header`, `Footer`, `Sidebar` для кабинета).
- Карточки уроков/тарифов.
- Модалки (для подтверждений).
- Компонент уведомлений (toast).

Стремиться к переиспользованию компонентов, а не копипасте верстки.

### 6. Интеграция AI‑ассистента на сайте (виджет)

На лендинге и/или в кабинете предусмотреть:
- Плавающую кнопку/иконку «Спросить AI» в правом нижнем углу.
- При клике открывается чат‑виджет:
  - список сообщений;
  - поле ввода;
  - индикация загрузки.
- Пока можно использовать мок‑ответы или простой `/api/ai` endpoint (будет реализован на бекенде позже).

### 7. Нефункциональные требования

- **Производительность:** Lighthouse не ниже 80 на десктопе.
- **SEO:** мета‑теги для публичных страниц, OpenGraph для лендинга.
- **Код‑стайл:** ESLint + Prettier (конфиг по умолчанию или по договоренности).
- **Структура проекта:** аккуратная, без «god‑components», разумное разбиение по модулям.
- **Язык интерфейса:** русский, с возможностью в будущем добавить en (вынести строки в отдельный файл/модуль).
